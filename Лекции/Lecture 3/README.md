## Part 1: Single Agent с TAO Loop (ReAct)

### Архитектура

Простейший агент: одна LLM + набор инструментов, работающий в цикле:

```
Thought → Action → Observation → Thought → ... → Final Answer
```

Это паттерн **ReAct** (Reasoning + Acting) — модель чередует вызов инструментов и рассуждения.

### Инструменты (τ-bench домен)

| Инструмент | Назначение |
|------------|-----------|
| `search_flights` | Поиск рейсов по маршруту и дате |
| `get_flight_details` | Детали конкретного рейса |
| `get_booking` | Информация о бронировании |
| `get_policy` | Политики авиакомпании (rebooking, cancellation, baggage) |
| `update_booking` | Изменение бронирования |

### Реализация на LangGraph

- `StateGraph` + `MessagesState` для управления состоянием диалога
- `ToolNode` для автоматического исполнения инструментов
- `parallel_tool_calls=False` — принудительно один инструмент за шаг (явный TAO цикл)
- Дополнительный LLM-вызов для извлечения THOUGHT, если модель генерирует только вызов инструмента без рассуждения

### Результаты тестирования

| Тест | Сложность | TAO циклов | Результат |
|------|-----------|-----------|-----------|
| Поиск рейсов | Простой | 1 | Корректный |
| Мульти-шаговый с явными инструкциями | Средний | 3 | Корректный |
| Естественный мульти-доменный запрос | Сложный | 4 | Частично некорректный — пропущены домены |

**Ключевая проблема:** один агент с множеством обязанностей = неполное покрытие на сложных запросах.

## Part 2: Multi-Agent Hierarchical System

### Принципы MAS

1. **Декомпозиция** — разбиение задачи на подзадачи
2. **Специализация** — каждый агент — эксперт в своём домене
3. **Коммуникация** — обмен структурированными данными
4. **Координация** — координатор управляет процессом

### Архитектура

**Coordinator (оркестратор) + 3 специализированных агента:**

| Агент | Инструменты | Зона ответственности |
|-------|------------|---------------------|
| `FlightAgent` | `search_flights`, `get_flight_details` | Поиск и сравнение рейсов |
| `PolicyAgent` | `get_policy` | Политики компании |
| `BookingAgent` | `get_booking`, `update_booking` | Управление бронированиями |

### Структурированная коммуникация (Pydantic)

- **`CoordinatorPlan`** — план декомпозиции: reasoning + список SubTask с приоритетами
- **`SubTask`** — подзадача: agent_name, description, priority
- **`AgentResult`** — результат агента: status, result, tools_used

### Цикл координатора

1. **Planning** — LLM с `structured_output` декомпозирует запрос в подзадачи
2. **Delegation** — подзадачи сортируются по приоритету и делегируются специалистам
3. **Synthesis** — результаты объединяются в единый ответ для клиента

### Результат

На том же сложном мульти-доменном запросе, где Single Agent провалился, MAS корректно:
- Декомпозировал задачу на 5 подзадач
- Получил информацию обо всех доменах (бронирование, рейсы, политики отмены и перебронирования)
- Синтезировал полный ответ с конкретными ценами и условиями

**Trade-off:** MAS использует больше LLM-вызовов и работает медленнее, но даёт более полные ответы на сложных запросах.

## Part 3: Critics и Quality Control

### Critic Agent

Один из 6 принципов MAS — **верификация**. Critic Agent:
- Анализирует результаты других агентов
- Находит ошибки, пропуски, нарушения политик
- Не действует — только оценивает

### Паттерн Reflexion ([Shinn et al., 2023](https://arxiv.org/abs/2303.11366))

```
Agent → Result → Critic → Approved?  ── Yes ──> User
                              │
                             No → Agent (retry with feedback)
```

### Реализация

**`CriticFeedback`** (структурированный вывод):
- `approved` — одобрен ли ответ
- `score` — оценка 0–10
- `issues` — найденные проблемы
- `suggestions` — предложения
- `reasoning` — обоснование

**`EnhancedCoordinator`** — координатор с интегрированным контролем качества:
- Шаг 4: Critic review — оценка ответа
- Шаг 5: Revision — если score < 7, ответ пересоздаётся с учётом фидбека

### Takeaways

- Critic добавляет верификацию, фидбек-цикл и измеримость качества
- Trade-off: +1–2 LLM-вызова (дополнительная латентность), но выше надёжность
- **Critic должен быть мощнее оцениваемого агента** (если агенты на nano, critic — на mini или выше)
- В критических доменах (финансы, здравоохранение, авиация) — **всегда** добавляйте Critic

## Part 4: Benchmark и сравнение архитектур

### Методология

Fact-based objective scoring — проверка наличия конкретных ожидаемых фактов в ответе (цены, сборы, номера рейсов, детали политик).

### Результаты

| Архитектура | Avg Score | Avg Time | Avg Tools |
|-------------|-----------|----------|-----------|
| Single Agent (ReAct) | 6.0/10 | 4.6s | 1.7 |
| Hierarchical MAS | 9.3/10 | 15.4s | 3.0 |
| MAS + Critic | 10.0/10 | 21.2s | 3.0 |

### Где архитектуры расходятся

- **Простые запросы** (1–2 инструмента): Single Agent быстрее при том же качестве
- **Multi-domain** запросы: MAS выигрывает благодаря гарантированной декомпозиции (2/10 → 10/10)
- **Tricky** запросы (ограничение по классу): только MAS + Critic ловит все нюансы (6/10 → 10/10)

### Рекомендация для продакшена

| Сценарий | Архитектура |
|----------|------------|
| Простые запросы (1–2 tools) | Single Agent — быстро, то же качество |
| Сложные запросы (3+ tools) | Hierarchical MAS — гарантированное покрытие |
| Критические домены | MAS + Critic — верификация качества |
