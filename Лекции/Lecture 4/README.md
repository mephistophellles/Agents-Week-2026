## Part 1: Agent

### Архитектура

ReAct-агент на LangGraph с 4 инструментами:

| Инструмент | Назначение | Тип |
|------------|-----------|-----|
| `get_booking` | Поиск бронирования по коду | Read-only |
| `search_flights` | Поиск доступных рейсов | Read-only |
| `change_booking` | Изменение рейса в бронировании | Write |
| `cancel_booking` | Отмена бронирования | Write |

### Ключевые концепции

- **Trajectory** — полная запись одного прогона: шаги (action, observation, response) + снапшоты БД до/после
- **Step** — атомарная единица траектории
- **DB snapshots** — `db_before` / `db_after` для state-based оценки

### Policies (правила поведения)

1. Запрашивать подтверждение перед изменением/отменой бронирования
2. Сначала получить текущее бронирование, потом предлагать изменения
3. Сообщать разницу в цене при изменении рейса
4. Не раскрывать данные других пассажиров
5. Если информации недостаточно — спросить пользователя

## Part 2: Task Basket

Хорошая корзина задач: разнообразные сценарии, разная сложность, крайние случаи, баланс.

### Формат задачи (расширенный)

- `query` — запрос пользователя
- `expected_state_changes` — ожидаемые изменения в БД (для state-grading)
- `policies_to_check` — какие политики особенно важны для задачи
- `category`, `difficulty`, `needs_dialogue` — метаданные для анализа

### Состав корзины

12 задач (10 базовых + 2 extra-hard), сбалансированных по:
- **Категориям:** info, search, change, cancel
- **Сложности:** easy, medium, hard, extra_hard
- **Типу:** single-turn vs multi-turn (needs_dialogue)

## Part 3: Criteria and Graders

Три комплементарных грейдера:

| Грейдер | Тип | Что проверяет |
|---------|-----|---------------|
| `state_grader` | Детерминированный (код) | Состояние БД до vs после прогона |
| `policy_grader` | Эвристический (код) | Нарушения политик по тексту траектории |
| LLM-as-Judge | LLM | Полезность, обоснованность, эффективность |

### State Grader

- Ключевой принцип: **оценивать по изменению состояния среды, а не по тексту ответа** (как τ-bench `calculate_reward`)
- `expected=None` → БД не должна измениться (read-only задача)
- Поддержка валидационных правил: `compare_time`, `compare_price` с операторами `gt`, `lt`, `min`, `max`

### Policy Grader

- Эвристическая проверка соблюдения поведенческих правил
- Keyword search по тексту траектории
- Проверки: подтверждение перед мутацией, get_booking перед change/cancel, упоминание цены

## Part 4: Run v1 — Single-Turn

Первый базовый прогон: агент отвечает один раз, без продолжения диалога.

### Проблемы

- Задачи, требующие диалога (`needs_dialogue=True`), проваливаются — агент задаёт уточняющий вопрос, но некому ответить
- State checks: **0/8 passed** для задач с ожидаемыми изменениями

**Вывод:** Single-turn не покрывает диалоговые задачи. Нужен Iron User.

## Part 5: Iron User — симулированный пользователь

### Концепция

**Iron User** — LLM, которая симулирует реалистичного пользователя:
- Знает цель задачи и контекст пользователя
- Отвечает как реальный пользователь (1–2 предложения)
- Сигнализирует завершение фразами «thank you», «great»

Заменяет дорогих человеческих тестировщиков для multi-turn оценки.

### Iron User v1 (naive) — демонстрация проблем

- Нет `max_steps` → может зацикливаться бесконечно
- Нет обработки ошибок в промпте → пользователь повторяет неудавшиеся запросы
- **Проблема API-дизайна:** `search_flights` возвращает `id` (SU2456), а `change_booking` требует `flight_key` (SU2456_0220) → агент не может угадать формат → зацикливание

> **Урок:** даже «умный» агент не может угадать формат API. Плохой API-дизайн → зацикливание.

### Iron User v2 (improved)

**Исправления:**
1. `search_flights` теперь возвращает `flight_key` — агент видит оба идентификатора
2. Промпт Iron User расширен: «если ассистент сообщает об ошибке — принять альтернативу или отказаться»
3. `max_steps=30` — защитный лимит от зацикливания
4. Расширенные end-фразы: «never mind», «ok fine» как сигналы завершения

## Part 6: Human Labeling

Перед автоматическим судейством — **ручная разметка выборки траекторий**.

### Критерии оценки

| Критерий | Описание |
|----------|----------|
| **Usefulness** | Решил ли агент проблему пользователя? |
| **Groundedness** | Агент утверждал только факты из БД? |
| **Efficiency** | Агент использовал разумное число шагов? |

Метки: `good` / `so-so` / `bad`

Человеческие метки = **ground truth** для измерения качества судьи (Cohen's κ).

## Part 7: LLM-as-Judge — итеративное улучшение

5 версий судьи, с измерением улучшения через **Cohen's κ** vs человеческих меток:

| Версия | Изменение | Ожидаемый прирост κ |
|--------|----------|-------------------|
| v1 | Naive: один промпт, все критерии, без определений | baseline |
| v2 | Конкретные определения уровней (good/so-so/bad) | +0.1–0.2 |
| v3 | Отдельные промпты по критериям + chain-of-thought | +0.1–0.15 |
| v4 | Few-shot примеры (Critique Shadowing) | +0.05–0.1 |
| v5 | Более сильная модель (gpt-4o) | +0–0.05 |

**Cohen's κ** корректирует на случайное согласие: κ=0 — не лучше случая; κ>0.6 — существенное согласие.

### Ключевые наблюдения

- Naive LLM-судья ненадёжен (κ ≈ 0.2–0.3)
- **Prompt engineering > model scaling** — v2→v3→v4 дают больше, чем v4→v5
- **Few-shot из человеческих меток** (Critique Shadowing) — самое большое единичное улучшение
- **Cost/quality tradeoff:** gpt-4o-mini + хорошие промпты ≈ gpt-4o + naive prompt при 17× меньшей стоимости

## Part 8: Summary

### Ключевые выводы

1. **Детерминированные грейдеры первыми** — state + policy проверки дёшевы и ловят очевидные провалы
2. **Iron User > single-turn** — симуляция диалога выявляет проблемы, невидимые в v1
3. **Качество судьи имеет значение** — naive LLM-судьи ненадёжны (κ ≈ 0.2–0.3)
4. **Prompt engineering побеждает scaling модели** — v2→v3→v4 дают больше, чем v4→v5
5. **Few-shot из человеческих меток** — «Critique Shadowing» — самое большое единичное улучшение
6. **Cost/quality tradeoff** — gpt-4o-mini + хорошие промпты ≈ gpt-4o + naive prompt при 17× меньшей стоимости
